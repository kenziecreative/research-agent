The plan-generator subagent produces a discovery strategy, the directory scaffold includes `research/discovery/`, and the CLAUDE.md template references the discover skill in the skills table, workflow, and enforcement sections.

<decisions>
## Implementation Decisions

### Discovery strategy generation
- **Per-phase channel map** — `discovery-strategy.md` contains a project-specific extract of the type-channel map, pre-matched to the actual phase names from the research plan
- **Generated by the plan-generator subagent** — the existing Sonnet subagent that produces `research-plan.md` also generates `discovery-strategy.md` in the same pass (it already has the research type, topic, and phase structure)
- **Strategy is authoritative** — if `research/discovery/strategy.md` exists, the discover skill reads it directly (pre-matched phases, no keyword guessing). Falls back to type-channel map + keyword matching only if no strategy file exists
- **Location:** `research/discovery/strategy.md` — inside the discovery directory alongside candidate files

### Directory scaffold changes
- **Seeded with strategy.md only** — `research/discovery/` is created and populated with `strategy.md` from the plan-generator. Candidate files appear later when the user runs `/research:discover`
- **Simple directory entry with comment** in CLAUDE.md's directory structure documentation — add `discovery/` with a one-line comment (e.g., `# Discovery strategy and candidate sources`), consistent with how other directories are documented
- **Verification checklist updated** — Step 5 verification adds `discovery/strategy.md` to the list of files confirmed during scaffold verification
- **No discovery reference files copied** — the discover skill reads playbooks and type-channel maps from the research-agent project's `.claude/reference/` at runtime. The scaffolded project only needs its own `strategy.md`

### CLAUDE.md template updates
- **New skills table row** — `/research:discover` gets its own row with trigger pattern and job description, consistent with existing table format
- **Brief workflow mention** — one line in the workflow section explaining that `research/discovery/strategy.md` maps phases to channels and the discover skill reads it
- **Optional recommendation, not mandatory** — enforcement section includes a soft rule: "Running `/research:discover` at the start of each phase is recommended but not mandatory. Sources can still be found manually and processed with `/research:process-source`"
- **Report lists 9 skills** — Step 6 (Report) updated from 8 to 9 skills, including `/research:discover`

### Workflow cycle integration
- **Discover is a substep of Collect** — the 5-step cycle (Collect→Connect→Assess→Synthesize→Verify) stays at 5 steps. CLAUDE.md notes: "Start Collect by running `/research:discover`, then process the candidates with `/research:process-source`"
- **Inline note on STATE.md Collect checkbox** — reads: "**Collect** — Sources gathered for this phase's questions (start with /research:discover)". Visible reminder without adding a separate checkbox
- **Start-phase mentions discover** — `/research:start-phase` output includes: "Recommended first step: run `/research:discover` to find candidate sources for this phase"
- **Phase-completion text unchanged** — no mention of discovery at completion time; start-phase and CLAUDE.md handle the recommendation

## Specific Ideas

- Strategy file is the "express lane" — scaffolded projects get pre-matched phase-to-channel mappings, while non-scaffolded projects (or projects with no init) fall back to keyword matching against type-channel maps
- The discover skill should check for `research/discovery/strategy.md` first, then fall back to type-channel map lookup — this priority order is established in Phase 3 but the init creates the file that enables the fast path
